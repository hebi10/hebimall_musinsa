## 작업 원칙
- 모든 답변과 설명은 한국어로 작성한다.
- 안정적인 수정, 기존 동작 보존, 유지보수성 향상을 우선한다.
- 추측보다 실제 코드, 스크립트, 서비스 흐름을 기준으로 판단한다.
- 사용자가 명시적으로 요청하지 않으면 커밋, 푸시, 배포하지 않는다.
- 비밀키, 토큰, `.env` 값은 출력하지 않는다.

## 문서 사용
- 작업 시작 전 `docs/README.md`를 먼저 확인한다.
- `docs/README.md`에서 이번 작업과 직접 관련된 문서만 읽는다.
- 허브에 없는 문서가 필요하면 새 문서를 만들고 `docs/README.md`에 항목을 추가한다.
- 동작, 구조, 명령, 정책이 바뀐 경우에만 관련 문서를 갱신한다.
- 문서 갱신은 작업 판단에 필요한 최신 내용 위주로 짧게 남긴다.

## 코드 작업
- 수정 전 관련 API, package.json 스크립트, 핵심 파일을 확인한다.
- 기존 구조, 네이밍, 권한 정책을 우선 보존한다.
- 요청 범위를 넘는 리팩터링과 관련 없는 파일 변경은 하지 않는다.
- 사용자가 만든 변경사항을 되돌리지 않는다.
- 삭제, 초기화, 강제 푸시, 대량 이동 같은 파괴적 작업은 먼저 확인한다.

## 검증
- 가능하면 변경 범위에 맞는 타입체크, 린트, 테스트, 빌드 중 적절한 검증을 실행한다.
- 검증 명령은 `package.json`에 정의된 실제 스크립트를 우선 사용한다.
- 검증을 실행하지 못한 경우 이유와 남은 리스크를 명확히 남긴다.
- 한국어 문구를 수정했다면 깨짐 여부를 간단히 확인한다.
